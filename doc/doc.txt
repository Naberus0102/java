
Authentication (Xác minh bạn là ai, có tài khoản chưa)
Nhập:
signup: username, password, name, email, phone number...
signin: username or email, password.

signup >> api đăng kí tài khoản >> server >> tạo user mới >> database
signin >> api đăng nhập >> server >> So sánh username or email, password >> database

Authorization (Xác minh bạn là ai, có quyền gì)
Server xác định xem ai đang thực hiện yêu cầu, người gửi có quyền thực hiện yêu cầu không
Diễn ra lặp lại suốt quá trình sử dụng ứng dụng >> làm sao để server biết được người gửi request là ai?
Một số cách không khả thi: 
Dùng userID (hacker có thể giả danh), 
Dùng userID và password lưu trong bộ nhớ tạm thời (Nguy cơ bị ăn cắp pasword), 
Bắt user login lại (bad UX)
Cách chọn: Server truyền về 1 mã bí mật cho client, khi thực hiện hành động thì client truyền lên cho server
Điểm yếu: Hacker có thể mạo danh bằng cách trộm mã bí mật >> Khắc phục: Đặt mã bí mật vào két sắt, chỉ server có chìa khóa
Điểm yếu: Hacker có thể trộm luôn cái két >> Khắc phục: Đặt thời gian hết hạn cho cái két (1-15p)

>>JWT (Json Web Token)
- Chỉ server mới có chìa để xác nhận
- Có thể chứa thông tin đã mã hóa của user như userID
- Có thời gian hết hạn

Theo ví dụ trên đoạn mã đặt vào két sắt gọi là Access token secret, két sắt gọi là access token
Vì access token có thể hết hạn >> cần tạo ra 1 thứ gì đó giúp người dùng đổi access token khi hết hạn
>> Server khi truyền access token thì truyền thêm 1 thứ song song có vai trò đó: Refresh token
Khi access token hết hạn >> client gửi lên server refresh token để tạo access token mới

Đặc điểm access token và refresh token:
Refresh token: Dùng để tạo access token mới, hết hạn lâu, không nhất thiết dùng JWT, lưu trong cookie và bảo mật nhiều lớp, lưu trong db (sesion), gửi qua cookie
Access token: Dùng để xác thực người dùng, hết hạn nhanh, lưu trong bộ nhớ tạm, không lưu trong db, gửi qua header (chống csrf)
